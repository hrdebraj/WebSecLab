01Spot the two sinks

Posting a comment stores author and content. Clicking Moderate triggers a second server-side query that uses one of those fields. Your first job: figure out which field is re-used and how.

02Follow the behaviour

Post two comments with different authors. Moderate each one. Observe the Moderation result panel — does its output depend on the author value, the content value, or both?
💡 Burp tip
In Burp HTTP history inspect the response of /sqli2/l2/moderate/<id>. You'll see an SQL string echoed (for teaching purposes) — but also observe how the COUNT and MAX change.

03Ladder the injection on author

Post comments with progressively trickier authors. Don't try UNION right away.
  1. Author = x → baseline
  2. Author = x' → does moderation error?
  3. Author = x'' → normal again?
  4. Author = x' OR '1'='1 → a boolean game
Each post creates a new comment you can moderate individually. Reset the lab between runs if the table gets too noisy.

04Confirm the query shape

Based on the error / row counts, write down what you believe the vulnerable query looks like.
💡 likely shape
SELECT COUNT(*) AS total, MAX(id) AS last_id FROM comments WHERE author='<here>'

05Pick a technique

You have two classic roads once you control a SELECT:
  • UNION-based: append a UNION SELECT that returns data from another table. The column count of the original query matters.
  • Error-based: force the database to raise an error whose text includes the value you want. MySQL makes this easy with functions like EXTRACTVALUE or UPDATEXML.
The outer query here uses COUNT(*), MAX(id) — how many columns, what types?

06Build a test payload (conceptually)

Before writing the final payload, sketch it: ' UNION SELECT <col1>, <col2> FROM users WHERE <pred>-- . What integer-castable value pair can you grab from users to place in col1 & col2?
💡 column-count gotcha
The outer SELECT lists two aggregate columns. Your UNION SELECT must also return exactly two columns. MySQL will allow non-numeric data in aggregate slots if the query doesn't use GROUP BY.

07Extract the target

The admin has a private_note. Design your payload to read that column. Post a comment whose author contains your payload; moderate that comment; read the response.
💡 what to read on success
Look at the Rows returned panel under "Moderation result". A properly-crafted UNION will list two rows: the original aggregate row AND your injected row.

08Why does parameterisation not save us?

The INSERT into comments is parameterised. Yet we still own the DB. Summarise in your own words how trust boundaries can be broken after data enters safe storage.
