01Model the flow on paper

Write down the data you control and where it lives:
  1. You POST an email. Server saves {email, token}.
  2. Server returns the token.
  3. You POST {token, new_password}. Server looks up the stored e-mail by token, then applies the password to whatever user owns that e-mail.
The second-order sink is somewhere in step 3.

02Ladder-probe the e-mail field

In step 1, submit progressively:
  1. A normal e-mail such as me@lab.test.
  2. One with a single quote: me'@lab.test.
  3. One with double single quotes: me''@lab.test.
Does step 1 ever error? (It won't — it's parameterised.) The store is silent. The breakage must come later.

03Probe via step 3

Request a token for me'@lab.test, then in step 2 submit that token with any new password. Observe the flash message. Is it generic-success, or is there an SQL error?
💡 what are we hoping to see?
An SQL error at step 3 strongly implies the server interpolates the stored e-mail into a new query — exactly the second-order pattern.

04Deduce the query shape

If the reset silently changes your password, the update is probably UPDATE users SET password='<new>' WHERE email='<stored>'. You control <stored> completely.

05Design the predicate

You don't know admin's e-mail address, but you can shape the WHERE predicate to match any row you like. Think of ways to make the predicate always true for the admin row and not match your decoy e-mail at all.
💡 predicate-rewriting hints
  • Terminate the string literal with a single quote.
  • Use OR with a condition that selects admin specifically — e.g. by role, username or id.
  • Discard the trailing quote with a comment sequence so the statement parses.

06Minimise blast radius

An all-true predicate would change every user's password. That's messy and detectable. Target only admin. After your request token / reset pair, verify with the admin login form.

07Execute

  1. Request a token for your crafted e-mail.
  2. Use the token to reset with a password you choose.
  3. Login as admin to capture the flag in their private note.
If verification fails, inspect the request with Burp Repeater and consider whether quoting / URL encoding of your payload mangled anything.

08Lesson

The safe INSERT turned the database into a trusted source of evil input. A defender must either (a) parameterise every query (including the UPDATE), or (b) apply schema-level validation (e.g. an e-mail regex CHECK) so hostile values never reach storage.

Request (safe)

cur.execute("INSERT INTO reset_tokens (email, token) VALUES (%s,%s)", (email, token))

Reset (vulnerable)

row = query("SELECT email FROM reset_tokens WHERE token=%s AND used=0", (token,), one=True)
stored_email = row["email"]
sql = f"UPDATE users SET password='{new_pw}' WHERE email='{stored_email}'"
raw_execute(sql)
