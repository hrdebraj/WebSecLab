01Understand the flow

There are two endpoints here that handle your input: /register stores it, and /change-password later uses it. In a classic 1st-order SQLi, the same request that carries the payload is also the one that runs the injected SQL. What makes this a second-order vulnerability? Think about where input goes first and where it is re-used later.

02Probe registration first

Register an ordinary account (e.g. test / test). In Burp Proxy → HTTP history, inspect the request. Does registration echo anything back? Does it error on special characters? Try registering a username containing a single quote '.
💡 what should I observe?
Registration is likely using a safe prepared statement. The single quote is stored verbatim. That means the username field can hold SQL metacharacters as pure data — which is fine, until someone interpolates it into a new query.

03Attack surface = the second query

Change your password. In Burp, inspect the request to /change-password. Notice how the server doesn't ask you for a username — it trusts the one you registered with. That username is fetched from the session and pasted into a new query.
💡 the likely query shape
UPDATE users SET password='<new_password>' WHERE username='<stored_username>'

04Ladder the injection

Before going for the kill, verify the injection point step-by-step. Register a user whose name contains a single quote, e.g. a'. Log in? — you can't, because the app logs you in automatically at registration. Now trigger change password — does the app error? That error (or lack of one) tells you the username is being concatenated.
💡 lad­dering technique
Classic probing order when you suspect string-context SQLi:
  1. Baseline — input without any metacharacter → works.
  2. Single quote ' → error or different behaviour.
  3. Two single quotes '' → works again (escaped).
  4. This confirms the input lands inside a quoted string literal.

05Reason about the WHERE clause

You control the value between the quotes in WHERE username='<here>'. What would you need to inject so that the UPDATE affects admin instead of your user? Remember you set your registered username once, forever, at step 02.
💡 shape of payload (no final answer)
Your stored username must terminate the string literal, then change the WHERE predicate so that the row matched is admin's. A single-line SQL comment (-- followed by a space, or #) can help you discard the trailing quote that the server will append.

06Pre-flight check

Use the Reset Lab button (sidebar) to clear any failed attempts before your final run. Then: (a) register with your crafted username + a throwaway password, (b) change password with the new admin password you want, (c) go to the verification form and log in as admin using your chosen password.

07Post-exploitation reflection

Why did a safe parameterised INSERT not protect the application? Write down two mitigations you would apply. Hint: parameterisation is about the query, not about the data. Defensive measures must follow the data wherever it is re-used.

Registration handler (safe)

cur.execute("INSERT INTO users (username, password, role) VALUES (%s,%s,'user')",
            (username, password))

Change-password handler (vulnerable)

stored_user = session["sqli2_l1_user"]   # from registration
sql = f"UPDATE users SET password='{new_pw}' WHERE username='{stored_user}'"
raw_execute(sql)

The defect: stored_user is treated as trusted internal data, but it originated from the attacker.
