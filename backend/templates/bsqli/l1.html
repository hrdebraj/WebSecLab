Level 1 · Boolean-Based Login Probe

A seemingly harmless "check if username is available" endpoint returns only {"exists":true} or {"exists":false}. No errors. No result rows leak. But the answer changes based on a query you control.
easy goal: extract admin's api_key burp: Repeater + Intruder

Source (relevant)

Username availability

Endpoint: GET /bsqli/l1/check?username=<u>. Response is JSON with a single boolean field.

(no request yet)

Target

The admin user has a column api_key that begins with FLAG{. Extract the full string using the availability endpoint as your only oracle.

01Confirm the injection point

Hit the endpoint with:
  1. username=alice → {"exists":true}
  2. username=zzz → {"exists":false}
  3. username=alice' → ?
  4. username=alice'' → ?
If request 3 returns false and request 4 returns true, you're inside a quoted string literal: WHERE username='<x>'.

02Craft the oracle

Design a value that closes the quote and plugs in a boolean expression: alice' AND <test> -- . If <test> is true, exists stays true; if false, it flips to false. That asymmetry is your oracle.
💡 sanity payloads
  • alice' AND 1=1-- → true
  • alice' AND 1=2-- → false
If both come back true, your injection doesn't parse. If both come back false, you broke the query.

03Pivot to another table

The target column is in users but on a different row. Use a subquery: <test> becomes something like (SELECT ... FROM users WHERE username='admin'). Your oracle fires only when the inner expression is truthy.

04Extract one character

Use SUBSTRING() (or MID()) plus either ASCII() or a direct equality comparison to test one character at a time. Start with the length: is the length of api_key greater than 0, 1, 2, ... ? Use binary search to save requests.
💡 discovery sequence
  1. Length discovery: LENGTH((SELECT api_key FROM users WHERE username='admin')) > N
  2. Character test: SUBSTRING((SELECT api_key ...),1,1) = 'F'
  3. Or binary: ASCII(SUBSTRING(...,1,1)) > 77

05Automate with Burp Intruder

Send the probing request to Intruder. Use Sniper mode with two positions: one for the index, one for the ASCII test value. Configure Grep - Match on the string "exists":true to classify responses.
💡 Intruder payload types
Use the Numbers payload type: first position iterates 1..60 (length), second iterates 32..126 (printable ASCII). Alternatively, write a Python helper outside Burp — many classrooms find requests clearer than Intruder for blind attacks.

06Assemble

Join all recovered characters. The flag starts with FLAG{ and ends with }.

07Reflection

A single boolean leak is enough to exfiltrate an entire table. What mitigation(s) would completely stop this? What mitigations only slow it down?

Endpoint (vulnerable)

@app.route("/bsqli/l1/check")
def bsqli_l1_check():
    username = request.args.get("username", "")
    sql = f"SELECT id FROM users WHERE username='{username}'"
    rows = raw_execute(sql)
    return jsonify({"exists": len(rows) > 0})
