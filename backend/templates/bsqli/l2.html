Level 2 · Time-Based Product Search

A product category API always responds the same way: {"status":"ok"}. No data. No errors. But the server does spend time executing your query.
medium goal: read products.secret_note for the 'accessories' row oracle: response time

Search by category

Endpoint: GET /bsqli/l2/search?category=<c>. The response body is always identical and does not disclose row data.

(no request yet)

The panel above shows the total wall-clock time of the round-trip. When you inject a SLEEP(n) that executes, the time jumps.

Target

One row in the products table has secret_note set to a flag. Normal users cannot see it. Extract it using timing differences.

01Confirm the injection point silently

The response body never changes. You can't rely on error messages or content diffs. What's left? Time.
  1. Try category=kitchen — baseline time.
  2. Try category=kitchen' AND SLEEP(0)-- — should match baseline.
  3. Try category=kitchen' AND SLEEP(3)-- — should add ~3 seconds.
If step 3 is slow, injection confirmed. If not, revisit quoting.
💡 how many rows fire SLEEP?
When you write SELECT ... WHERE category='kitchen' AND SLEEP(3), MySQL evaluates SLEEP(3) once per matching row. That can turn a 3-second delay into 15. Aim your payload so it runs the sleep exactly once.

02Make the oracle conditional

You want:
"If my test is true, sleep; if false, don't."
The classic pattern uses IF(cond, SLEEP(n), 0) in MySQL. Wrap your boolean inside and observe the delay.

03Pin the sleep to a single row

To avoid multiplying the delay, scope your condition so only a constant returns from the outer query. Two common tricks:
  • Make the outer predicate match zero or one row (AND id=1).
  • Or wrap everything in a UNION SELECT IF(cond, SLEEP(3), 0) with a predicate that matches exactly one row.

04Find the flag row first

You don't know which row holds the flag. Use the oracle to find it:
  • Is there a row where secret_note LIKE 'FLAG%'?
  • What is its id? Binary-search through the id range.
  • What is LENGTH(secret_note) for that row?

05Character-by-character extraction

Now loop: for each position i from 1 to length, test SUBSTRING(secret_note, i, 1) = '?' for every printable ASCII byte — or binary-search via ASCII(SUBSTRING(...)). Sleep for, say, 2 s on match and 0 on miss. Don't go below 1 s because the DB has noise.
💡 scripting the loop
This is easier in a small Python script than in Burp Intruder because Burp's default wait thresholds aren't tuned for binary searching. Use Python requests + time.perf_counter() and set a clear threshold (e.g. response > 1.5s ⇒ true).

06Stabilise against jitter

Containers introduce noise. Repeat each probe 2-3× and take the minimum, or use a sleep long enough that noise cannot misclassify. Too short → false negatives. Too long → slow extraction.

07Reflection

Time-based SQLi is the oracle of last resort — slow but universal. A defender can detect it via abnormally long queries or via WAF rules matching on SLEEP/BENCHMARK. What would you do at application level to make it impossible?

@app.route("/bsqli/l2/search")
def bsqli_l2_search():
    cat = request.args.get("category", "")
    sql = f"SELECT name, price FROM products WHERE category='{cat}'"
    start = time.time()
    try:    raw_execute(sql)
    except: pass
    elapsed = time.time() - start
    return jsonify({"status": "ok", "server_elapsed_ms": int(elapsed * 1000)})
