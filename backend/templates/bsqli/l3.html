Level 3 · Cookie-Based Tracking

The app sets a track cookie and pings /bsqli/l3/ping every page-load. The ping's body is empty. The only signal is the X-Status response header: OK or UNKNOWN.
hard goal: extract admin's password from users table channel: header-based oracle burp: Proxy + Repeater mandatory

Source (relevant)

Tracking ping

On first visit, the server sets track=tk_alice. Every hit of /bsqli/l3/ping looks up the cookie value in a tracking table and returns an empty response (HTTP 204) with either X-Status: OK (row found) or X-Status: UNKNOWN (nothing matched). 

(no request yet)

Because this is a header-only oracle, the JS fetch above will only see the status code. Send the request through Burp to read the X-Status header.

Target

Extract the password column of the admin user from the users table — not the tracking table. You'll need a cross-table blind read.

01Open Burp and intercept the ping

Browse to this page with the Burp proxy on. In HTTP history locate the request to /bsqli/l3/ping. Notice the Cookie: track=tk_alice header and the response header X-Status: OK. Send to Repeater.

02Find the injection context

In Repeater modify the cookie value to:
  1. tk_alice → X-Status: OK
  2. tk_aliceZZZ → UNKNOWN
  3. tk_alice' → ?
  4. tk_alice'' → ?
Cookie single-quote probing reveals a string-literal context. Burp's cookie editor is the cleanest way to send metacharacters without URL encoding.

03Build the boolean oracle

Swap the cookie to tk_alice' AND 1=1-- (true) and tk_alice' AND 1=2-- (false). Observe OK vs UNKNOWN.
💡 trailing spaces matter
MySQL's line comment is -- (two dashes followed by a whitespace) or #. In cookie headers a single space is fine.

04Cross the table boundary

The current query hits tracking, not users. Your condition must reference the users table through a subquery: (SELECT ... FROM users WHERE username='admin').

05Length then content

Get the password length first (binary search). Then pull one character at a time using SUBSTRING + ASCII. Each "byte" is one HTTP request whose only evidence is a response header — that's the point of the exercise.

06Automate (two options)

  • Burp Intruder: send the Repeater request to Intruder, set the cookie value as the attack position, use Cluster bomb for (index, ascii) pairs, and in Options → Grep-Extract extract the X-Status header so you can sort results.
  • Python: requests.get(url, cookies={'track': payload}) and check r.headers['X-Status']. Much faster for long extractions.

07Verify

Once you have the password, log in through /login as admin. If login succeeds you have completed the level — the flag for this level is the plaintext password you recovered.
💡 can't log in?
Double-check case sensitivity and special characters. The password contains a mixed-case string with at least one digit and one punctuation mark.

08Reflection

Blind signals can hide in almost anywhere: headers, cache-control, cookies, error pages that differ only in a whitespace. Defenders must assume any attacker-observable difference is an oracle.

Endpoint (vulnerable)

@app.route("/bsqli/l3/ping")
def bsqli_l3_ping():
    token = request.cookies.get("track", "")
    sql = f"SELECT username FROM tracking WHERE token='{token}'"
    rows = raw_execute(sql)
    resp = make_response("", 204)
    resp.headers["X-Status"] = "OK" if rows else "UNKNOWN"
    return resp
