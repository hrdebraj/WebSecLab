Level 3 · Strict Allow-list

The developer got serious. User input comes in via the URL fragment (#name=...), and only a small allow-list of tags (b, i, u, a) plus a single attribute (href) is permitted.
hard goal: execute JS despite an allow-list technique: protocol smuggling / DOM clobbering

Format: #name=<display_name_with_simple_html>. For example: #name=Hello%20%3Cb%3Eworld%3C/b%3E.

Rendered bio:

Filter rules

  • Only tags <b>, <i>, <u>, <a> survive.
  • All attributes are stripped except href on <a>.
  • Event-handler attributes (on*) are always removed.

See Source for the exact implementation.

01Read the filter carefully

Before firing payloads, understand exactly what the filter permits and forbids. Open Source. Write down the surviving tags, the surviving attributes, and the forbidden substrings/patterns.

02Event handlers are dead

The obvious <img onerror> trick from Level 2 is off the table: - <img> itself is stripped - on* attributes are stripped You have to find execution via a permitted construct.

03The <a href> attack surface

<a> with href is the widest opening. Which URL schemes are dangerous?
  • http:, https: — benign navigation.
  • data: — navigation, may execute scripts (depends on browser).
  • javascript: — executes on click. 🎯
💡 click requirement
javascript: URLs require the user (you) to click the link to execute. In a real attack you'd social-engineer the victim; in this lab, just click.

04Construct the payload

Build a hash like: #name=Click <a href="<scheme>?alert_here">here</a>. Escape the spaces and the colon if the filter is picky. Test by clicking the rendered link.
💡 common filter tricks
If the filter looks at href.startsWith('javascript:'), you can try upper-case (JaVaScRiPt:), leading whitespace, or &#x6a;-style HTML numeric character references that the HTML parser decodes before the filter runs on the attribute value.

05Capture the flag

Successful execution of revealFlag() (or alert(window.FLAG)) pops the flag card. If your javascript: URL strips, try:
  • character-entity encoding of the scheme
  • mixed case
  • control-character insertion such as &#9; (tab) between letters of the scheme

06Alternate roads (bonus)

  • DOM clobbering: create HTML that shadows a JS identifier used by the page (e.g. <a id="config">). Worth reading about even if this level doesn't strictly need it.
  • Mutation XSS: some sanitisers operate on a string but the browser later mutates the parsed tree. Not in scope here but a great rabbit hole.

07Reflection

Even a strict allow-list can be escaped if you leave a high-power primitive in the allow-list (href). A hardened filter must further constrain URL schemes to a safe allow-list (http, https, mailto).

Vulnerable script (inline)

const ALLOWED_TAGS  = new Set(['B','I','U','A']);
const ALLOWED_ATTRS = new Set(['href']);

function sanitize(root) {
  root.querySelectorAll('*').forEach(el => {
    if (!ALLOWED_TAGS.has(el.tagName)) { el.remove(); return; }
    [...el.attributes].forEach(a => {
      // strip everything not in ALLOWED_ATTRS and any on*
      if (!ALLOWED_ATTRS.has(a.name) || /^on/i.test(a.name)) {
        el.removeAttribute(a.name);
      }
    });
  });
}

function render() {
  const p = new URLSearchParams(location.hash.slice(1));
  const name = p.get('name') || '';
  const tmp = document.createElement('div');
  tmp.innerHTML = name;          // parse user HTML
  sanitize(tmp);                 // remove unsafe tags/attrs
  document.getElementById('bio').replaceChildren(...tmp.childNodes);
}

The sanitiser does not restrict the value of href — any URL scheme is permitted.
